Why does this platform exist?

Mid-to-large retailers operate POS terminals, regional distribution centres, and supplier networks as independent silos. By the time a stockout is visible on a dashboard it has already cost a sale. Replenishment decisions are made reactively, on stale data, by planners looking at spreadsheets rather than live demand signals. SmartRetail closes that loop.

⚠️ The Problems
Stockouts & overstock co-exist

Popular SKUs run dry while slow-moving stock ties up capital — because replenishment uses the same fixed lead-time rules for both.

Reactive, not predictive

Planners receive alerts after stock has already fallen below the reorder point. The replenishment cycle begins too late to prevent a customer-facing gap.

Disconnected systems

POS transactions, warehouse inventory, demand forecasts, and supplier lead times sit in separate databases with no real-time event flow between them.

✅ The SmartRetail Approach
Real-time POS ingestion

Every checkout event streams through Kinesis Data Firehose into the Sales Ingestion Service within seconds. Inventory positions update automatically.

ML-driven demand forecasting

SageMaker DeepAR produces P10/P50/P90 forecast bands per SKU per DC. The Replenishment Engine uses P90 to propose purchase orders proactively.

Automated replenishment workflow

Low-value orders auto-approve; high-value orders surface in the SC Planner console for one-click approval. Supplier notifications fire on EventBridge.

📈 End-to-end Platform Flow
  1. 1 POS terminal fires a checkout event → Kinesis Data Firehose batches and delivers to the Sales Ingestion Service (SIS) via API Gateway.
  2. 2 SIS persists the event and publishes SalesTransactionProcessed to EventBridge. IMS and DFS consume this asynchronously.
  3. 3 IMS decrements on-hand inventory. When stock falls below the reorder point it raises an InventoryAlertEvent.
  4. 4 Replenishment Engine (RE) receives the alert via FIFO SQS and automatically creates a Purchase Order in PENDING_APPROVAL state.
  5. 5 SC Planner reviews the PO in the console. Approve → RE transitions to APPROVED, notifies the Supplier Integration Service (SUP).
  6. 6 ARS aggregates metrics in real time across all schemas, feeding the Executive Dashboard and SC Planner dashboards with MAPE trends, OTD rates, and stockout risk indicators.

Production Architecture

Seven bounded-context services on ECS Fargate, connected by EventBridge and SQS. All traffic enters through API Gateway backed by a VPC-linked NLB.

🔒 Security Model
  • ● Cognito JWT validated at API Gateway authoriser and again inside each service (defence in depth)
  • ● IAM task roles per ECS service — least-privilege events:PutEvents, sqs:SendMessage, rds-db:connect
  • ● VPC Interface Endpoints for ECR, SQS, EventBridge, Secrets Manager — zero internet egress from services
  • ● Isolated subnets for RDS — no route table entry to internet or NAT
  • ● Secrets Manager for DB password — never in environment variables
🏛 Database Schema Isolation

Each service owns exactly one PostgreSQL schema and may only read/write rows within it. Cross-schema SQL JOINs are forbidden.

ARS reads multiple schemas through separate queries merged in Java using virtual-thread parallelism — never a multi-schema JOIN.

⚠️ This rule is enforced by ArchUnit tests. A cross-schema SQL JOIN or an import of software.amazon.* in a domain package will fail the Maven build.

Why a separate demo environment?

The demo is not a simplified mock. It runs the same service code and the same pipeline topology as production on real AWS infrastructure — Firehose ingestion, the SageMaker forecasting pipeline, and the post-processing Lambdas are all deployed. What changes is sizing and durability, not shape. Three constraints drive that right-sizing for a 1–2 day prototype demonstration.

💰 1 — Cost

A production deployment runs ~$850/month 24×7: Multi-AZ RDS (r6g.large), three NAT Gateways, multiple VPC interface endpoints, RDS Proxy, 7 ECS services at full size, and a nightly SageMaker training + batch-transform job. That is not justified for a 1–2 day prototype. The demo deploys the same pipeline topology for ~$80/month if left running 24×7 — and because it is ephemeral and torn down after each session, a paused 2-day run costs ~$5 (make demo-stop overnight, make demo-destroy when done).

🧠 2 — ML Training Data

The SageMaker DeepAR demand forecasting model requires months of real historical POS transaction data specific to the retailer's SKUs, store locations, and seasonal patterns. That data does not yet exist — this is a prototype. Forecast outputs in the demo are represented by pre-seeded placeholder values (Flyway migration V7) that stand in for what a trained model would produce once live transaction data has been collected over a sufficient window. The SageMaker pipeline, its execution role, and the S3 buckets are deployed in demo but dormant: the EventBridge schedule is disabled (enabled: false) and the trigger Lambda is throttled to zero (reservedConcurrentExecutions: 0), so standing ML cost is $0 — charges only accrue if a run is started manually.

⏱ 3 — Right-sized, not stripped down

The demo deploys the full ingestion + ML topology — Kinesis Firehose → API Gateway → SIS, the S3 events & SageMaker buckets, the batch-post-processor and ml-trigger Lambdas, and the SageMaker pipeline definition — so the architecture is demonstrable end to end. What changes is sizing and durability: a default VPC (no NAT), a single-AZ t4g.micro with no RDS Proxy, one ARM64 Fargate task per service, and ML compute gated off. The result bootstraps from cold in ~15 minutes (make demo-full-deploy) instead of the ~30–45 min a Multi-AZ, multi-NAT production stack needs.

🎯 Why each demo config choice
Demo choice Rationale
Default VPC, no NAT Gateway Public subnets + public-IP tasks reach ECR/S3 directly. Saves ~$33/NAT-month and several minutes of provisioning. HA and egress isolation are irrelevant for a throwaway stack.
t4g.micro, single-AZ, no RDS Proxy, no backups One presenter's load fits a micro; a single task per service creates no connection-pool pressure that a proxy would relieve. Data is reproducible from Flyway seed (V7), so backups and Multi-AZ add cost with no benefit.
ARM64 (Graviton) Fargate, 1 task / service ~20% cheaper than x86 at equal performance. One task per service covers single-presenter throughput; min/max 1/2 absorbs a restart.
SageMaker pipeline deployed but dormant Proves the ML wiring is real (role, buckets, pipeline definition) at $0 standing cost. No real POS history exists to train on yet, so the cron stays off and forecasts use seed data.
Lambdas outside the VPC With no NAT, a VPC-bound Lambda couldn't reach the S3 or SageMaker APIs. Running them outside the VPC keeps the S3 → Lambda → DFS path reachable via the API Gateway URL.
Firehose → public API GW (not VPC Link) Same delivery contract and SIS ingest endpoint (/v1/ingest/events) as prod, but targets the public NLB so no private networking is required.
RemovalPolicy.DESTROY everywhere Ephemeral by design — make demo-destroy leaves nothing behind, with no manual S3/ECR cleanup.

🙷 Hexagonal Architecture (Ports & Adapters)

Every ECS service follows the same package structure. The domain core is a pure Java model with zero AWS imports. Inbound adapters (REST controllers, SQS listeners) call into the domain through inbound ports. The domain calls outbound ports; JDBC repositories, EventBridge publishers, and SQS senders implement them as infrastructure adapters.

⚙️ Key Engineering Patterns
Pattern Mechanism Where applied
Idempotency SHA-256(transactionId) → sales.idempotency_keys (RDS); SELECT-then-INSERT in one transaction; 48h TTL via hourly @Scheduled purge SIS ingest endpoint
Optimistic locking WHERE id = :id AND version = :v on every PO UPDATE; 0 rows updated → HTTP 409 Conflict; prevents concurrent approval conflicts RE purchase orders
Async event ordering FIFO SQS for RE alert queue; messageGroupId = dcId ensures all alerts for a distribution centre are processed in arrival order RE alert consumption
Dead-letter policy maxReceiveCount=3; after 3 delivery failures message moves to DLQ; 14-day retention for investigation; alarms on DLQ depth All SQS queues
Virtual-thread parallelism Executors.newVirtualThreadPerTaskExecutor(); ARS fires up to 9 JDBC queries concurrently; joins with CompletableFuture.allOf(); no ForkJoin pool starvation ARS dashboard use cases
Contract-first API OpenAPI YAML edited first; mvn generate-sources produces Java server stubs; npm run generate-api produces TypeScript client; generated code is in .gitignore All 7 services + MFEs
Structured logging JSON per line; mandatory correlationId + traceId fields; X-Correlation-ID header propagated through entire call chain; PII never logged All services
State machine in RDS PO state: PENDING_APPROVAL → APPROVED → CANCELLED or REJECTED; no Step Functions — transitions are single-row UPDATEs with version check RE replenishment schema
